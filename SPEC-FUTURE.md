# Aegis — Future Layers & Planned Features

**Status:** Design document — not yet implemented
**Parent:** [SPEC.md](SPEC.md) (Layer 1, active)
**Last updated:** March 2026

This document specifies features that are designed but not yet built. Layer 1 (deterministic dispatch, dispatch state, runtime adapters, Olympus MVP) is the current implementation focus. These features are prioritized roughly in the order listed.

---

## 1. Metis: Natural Language Interpreter (Layer 2)

Metis is the conversational interface between the user and Layer 1. It activates when the user sends a message through Olympus's command bar in Steer or Ask mode.

**Prerequisite:** Olympus MVP with command bar must be built first. Direct-mode commands already cover all steering actions — Metis adds natural-language resolution on top.

### 1.1 Design

**Model:** Claude Haiku (or equivalent cheap/fast model). Target cost: < $0.001 per interaction.

**System prompt (~500 tokens):** Describes the orchestrator's available actions as a JSON schema. Includes current swarm state summary (active agents, beads queue size, active filters). Instructs the model to output a JSON array of actions.

**Available action types:**
```typescript
type SteerAction =
  | { type: "pause" }
  | { type: "resume" }
  | { type: "auto_on" }
  | { type: "auto_off" }
  | { type: "scale"; concurrency: number }
  | { type: "focus"; filter: string }
  | { type: "clear_focus" }
  | { type: "scout"; issue_id: string }
  | { type: "implement"; issue_id: string }
  | { type: "review"; issue_id: string }
  | { type: "process"; issue_id: string }
  | { type: "rush"; issue_id: string }
  | { type: "kill"; agent_id: string }
  | { type: "restart"; issue_id: string }
  | { type: "tell_agent"; agent_id: string; message: string }
  | { type: "tell_all"; message: string }
  | { type: "add_learning"; domain: string; text: string }
  | { type: "dispatch_oracle"; target: string; note: string }
  | { type: "reprioritize"; issue_id: string; priority: number }
  | { type: "summarize" }
  | { type: "noop"; explanation: string };
```

### 1.2 Example Interactions

| User input | Mode | Interpreted actions |
|------------|------|-------------------|
| "scout aegis-001" | Direct | `scout` with issue_id (no LLM needed) |
| "implement that auth issue" | Steer | `implement` (Metis resolves "that auth issue" to an issue ID from current beads state) |
| "review the last 3 closed issues" | Steer | Multiple `review` actions |
| "process the backlog" | Steer | `auto_on` |
| "stop auto" | Direct | `auto_off` (no LLM needed) |
| "the auth module seems like a mess" | Steer | `dispatch_oracle` for auth + `add_learning` |
| "kill that stuck worker" | Steer | `kill` (Metis identifies stuck agent from swarm state context) |
| "how's it going?" | Ask | `summarize` (read-only, no state changes) |
| "scale up, there's a lot of work" | Steer | `scale` with increased concurrency |

The interpreter LLM call includes the current swarm state (list of active agents, their issue assignments, queue depth) so it can resolve references like "that stuck worker" to a specific agent ID.

### 1.3 Olympus Integration

The command bar in Olympus gains a mode selector dropdown:
- ⚡ **Direct** — Bypasses LLM, pattern-matches deterministic commands (existing behavior)
- 💬 **Steer** — Routes to Metis (Haiku → structured actions)
- 🔍 **Ask** — Routes to read-only Metis call (summarize state, no actions taken)
- 📋 **Plan** — Routes to Prometheus (see Section 2)

Each mode implicitly selects the cost tier and model. The user never needs to choose a model directly.

### 1.4 Implementation Notes

- Metis is a single `metis.ts` module
- Takes user input string + current `SwarmState` → returns `SteerAction[]`
- Uses the cheapest available model (Haiku-class) via the runtime adapter
- Stateless — no conversation history between invocations
- If Metis returns a `noop`, display the explanation to the user without taking action

## 2. Prometheus: Strategic Planner (Layer 3)

Prometheus activates only when the user sends a message in Plan mode. It is the most expensive operation and runs at most once per invocation.

### 2.1 Design

**Model:** Claude Sonnet 4.6 by default, Opus 4.6 available for complex planning. Configurable.

**Purpose:** Decompose high-level goals into concrete beads issues with dependency chains. The planner has access to:
- The user's input (PRD, feature description, refactoring goal, etc.)
- A codebase summary (generated by an Oracle or read from a pre-existing overview file)
- Current beads state (`bd list --json`)
- The learnings store (Mnemosyne)

**Output:** The planner's response is parsed into `bd create` and `bd update` commands that the orchestrator executes directly. The planner does not dispatch agents — it populates beads, and Layer 1's normal dispatch loop picks up the new issues.

### 2.2 Example

User types in Plan mode: "We need to migrate from REST to GraphQL for the user service."

The planner reads the codebase, understands the current REST implementation, and emits a series of beads issues:
1. "Define GraphQL schema for User type" (P1)
2. "Create resolvers for user queries" (P1, blocked by #1)
3. "Create resolvers for user mutations" (P2, blocked by #1)
4. "Update client to use GraphQL" (P2, blocked by #2 and #3)
5. "Remove old REST endpoints" (P3, blocked by #4)
6. "Update integration tests" (P2, blocked by #2 and #3)

Layer 1 sees the new issues on its next poll and begins dispatching Oracles.

### 2.3 Implementation Notes

- Prometheus is a single `prometheus.ts` module
- Takes user input string + codebase context + beads state → returns a list of issue creation/update commands
- Uses an expensive model (Sonnet/Opus) — cost is displayed to the user before execution
- The orchestrator confirms with the user before creating the issues ("Prometheus proposes N issues. Create them?")
- No automatic dispatch — issues enter the normal beads queue

## 3. Configurable Dispatch Pipelines

Make the dispatch stage sequence user-configurable instead of the hardcoded Oracle → Titan → Sentinel pipeline.

### 3.1 Design

Users define pipeline templates in `.aegis/config.json`:

```json
{
  "pipelines": {
    "default": ["oracle", "titan", "sentinel"],
    "chore": ["titan"],
    "trivial": ["titan", "sentinel"],
    "security": ["oracle", "titan", "sentinel", "security-review"]
  }
}
```

The dispatch state machine becomes generic: for each issue, it looks up the pipeline based on issue type (or a user-specified override), then walks the configured sequence. `triage.ts` reads the pipeline config to determine the next stage for each issue.

### 3.2 Custom Stages

Each stage in a pipeline maps to a caste config:

```json
{
  "stages": {
    "oracle": { "caste": "oracle", "model": "claude-haiku-4-5" },
    "titan": { "caste": "titan", "model": "claude-sonnet-4-6" },
    "sentinel": { "caste": "sentinel", "model": "claude-sonnet-4-6" },
    "security-review": {
      "caste": "sentinel",
      "model": "claude-opus-4-6",
      "prompt_override": "Focus specifically on security implications..."
    }
  }
}
```

This allows the same caste (e.g., sentinel) to appear multiple times with different models and prompts.

### 3.3 Prerequisites

- Dispatch state redesign must be complete (stages are generic, not hardcoded)
- Runtime adapter layer must be in place (different stages can use different runtimes)

## 4. Mixed-Model Swarms

Dispatch different agent castes to different runtimes in the same swarm.

### 4.1 Use Case

Oracles and Sentinels run on local Qwen 2.5 Coder 32B via Ollama (cheap, good enough for read-only analysis), while Titans run on Claude Sonnet via cloud API (needs strong coding capability).

### 4.2 Config

```json
{
  "models": {
    "oracle": "ollama:qwen2.5-coder:32b",
    "titan": "anthropic:claude-sonnet-4-6",
    "sentinel": "ollama:qwen2.5-coder:32b",
    "metis": "ollama:qwen2.5-coder:32b",
    "prometheus": "anthropic:claude-sonnet-4-6"
  }
}
```

The provider prefix (before the colon) determines which `AgentRuntime` adapter is used. The model identifier (after the colon) is passed to the adapter.

### 4.3 Requirements

- `OllamaRuntime` adapter implementing `AgentRuntime` interface
- Budget tracking normalized to USD across all runtimes (Ollama models may report tokens but have zero dollar cost — track tokens uniformly, estimate cost as $0 for local models)
- Prompt templates may need runtime-specific adjustments (Ollama models may not support the same structured output features as Anthropic models)

### 4.4 Prerequisites

- Runtime adapter layer must be complete
- At least one non-Pi runtime adapter implemented

## 5. RAG-Based Mnemosyne Retrieval

Replace keyword-based domain matching in `mnemosyne.filter()` with semantic similarity search.

### 5.1 Motivation

When the Mnemosyne store exceeds ~200 records, keyword filtering on the `domain` field becomes unreliable. An issue about "JWT refresh tokens" won't match `domain="auth"` unless the keywords overlap exactly.

### 5.2 Design

- Generate embeddings for each learning's `text` field using a local model via Ollama
- Store embeddings alongside JSONL records (separate `.aegis/mnemosyne-embeddings.json` or inline)
- On retrieval: embed the issue title + description, compute cosine similarity against stored embeddings
- Return top-K results within the configured token budget
- Fallback to keyword matching if Ollama is unavailable

### 5.3 Prerequisites

- Ollama runtime adapter (or any local embedding model support)
- Mnemosyne store at a scale where keyword matching demonstrably fails (~200+ records)

## 6. Olympus Post-MVP Features

These extend the Olympus MVP (see SPEC.md §7) with additional panels and capabilities.

### 6.1 Beads Board (Kanban)

A kanban-style view of the beads issue lifecycle:

**Columns:** Ready → Scouting → Implementing → Reviewing → Done

Each issue card shows: ID, title, priority badge, type badge, and an indicator of which agent (if any) is currently assigned. Dependency lines are drawn between cards where `blocked-by` relationships exist.

Clicking an issue card opens a detail overlay showing: full description, Oracle assessment, Sentinel verdict, dependency graph, and which agents have worked on it.

### 6.2 Cost & Performance Panel

Recharts-powered visualizations:
- Running cost: line chart of cumulative cost over time
- Cost by caste: stacked bar chart
- Token burn rate: line chart of tokens/minute
- Per-issue cost: table with cost attributed to each beads issue

### 6.3 Event Timeline & Mnemosyne Panel

Combined sidebar with two tabs:
- **Timeline:** Scrolling log of all orchestrator events with timestamps
- **Mnemosyne:** List of all learnings, filterable by domain and type, with inline add

### 6.4 Agent Terminal (xterm.js)

Click-to-expand on agent cards shows full streaming terminal output via xterm.js, displaying the agent's complete session in real-time. Requires forwarding raw session output through SSE, which adds bandwidth overhead — opt-in per agent.

### 6.5 Steer/Plan/Ask Modes in Command Bar

Once Metis and Prometheus are implemented, the command bar gains mode selector dropdown with Steer, Plan, and Ask modes in addition to Direct.
